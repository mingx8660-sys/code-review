# Code Review — AI代码审查助手 🦐

当用户请求代码审查时，遵循以下流程：

## 审查流程

### 第1步：识别语言和技术栈
- 根据文件扩展名、代码特征识别语言
- 识别框架和依赖（React/Vue/Django/Flask等）

### 第2步：五维度审查

对每一段代码，逐项检查：

#### 1️⃣ 正确性 (Correctness)
- 逻辑错误：边界条件、空值处理、类型错误
- 并发问题：竞态条件、死锁、线程安全
- 异常处理：try-catch覆盖是否完整，异常吞没
- 资源泄漏：文件句柄、数据库连接、内存

#### 2️⃣ 安全性 (Security)
- 注入攻击：SQL注入、XSS、命令注入
- 认证授权：权限校验是否缺失
- 敏感信息：硬编码密码/密钥/Token
- 输入验证：用户输入是否经过校验和清理

#### 3️⃣ 性能 (Performance)
- 算法复杂度：是否有不必要的循环、重复计算
- 内存使用：是否有内存泄漏、大对象持有
- I/O操作：是否频繁读写、N+1查询
- 缓存策略：是否应该缓存未缓存

#### 4️⃣ 可读性 (Readability)
- 命名规范：变量/函数/类命名是否清晰
- 代码结构：函数是否过长（超过50行需要拆分）
- 注释质量：是否需要/过多/过时注释
- 设计模式：是否使用过当的模式

#### 5️⃣ 可维护性 (Maintainability)
- 重复代码：DRY原则是否被违反
- 耦合度：模块间依赖是否合理
- 测试覆盖：关键逻辑是否有测试
- 配置硬编码：魔法数字/字符串

### 第3步：输出审查报告

使用以下格式输出：

```markdown
## 📋 Code Review 报告

**文件**: `xxx.py`
**语言**: Python 3.12
**严重程度**: 🔴 严重 / 🟡 中等 / 🟢 建议

---

### 🔴 严重问题 (必须修复)

1. **[安全] SQL注入风险** — 第42行
   ```python
   cursor.execute(f"SELECT * FROM users WHERE id = {user_input}")
   ```
   **建议**: 使用参数化查询
   ```python
   cursor.execute("SELECT * FROM users WHERE id = ?", (user_input,))
   ```

### 🟡 中等建议 (建议修复)

2. **[性能] 不必要的循环嵌套** — 第15-25行
   ...

### 🟢 小建议 (锦上添花)

3. **[可读性] 魔法数字** — 第7行
   ...
```

### 第4步：评分

给出总体评分（10分制）和一句话总结：

```
总体评分: 7.5/10
总结: 代码逻辑清晰，但存在安全隐患，建议修复SQL注入问题后上线。
```

## 扩展审查维度

除了五维度审查外，根据代码类型自动启用以下专项审查：

### 6️⃣ Docker/容器化
- 镜像构建：多阶段构建、层优化、`.dockerignore`
- 安全性：root用户、CVE扫描、敏感信息泄露
- 运行时：资源限制、只读文件系统、健康检查
- 详见 `references/docker-checklist.md`

### 7️⃣ CI/CD 流水线
- 安全性：第三方Action固定版本、Secrets泄露、令牌权限
- 效率：缓存策略、并行构建、条件触发
- 流程：质量门禁、环境隔离、回滚机制
- 详见 `references/cicd-checklist.md`

### 8️⃣ 数据库/SQL
- 查询性能：N+1查询、索引使用、深分页
- 索引设计：复合索引顺序、覆盖索引、索引列函数操作
- Schema：字段类型、约束、迁移回滚、软删除
- 详见 `references/database-checklist.md`

### 9️⃣ API 设计
- RESTful规范：URL命名、HTTP方法、状态码
- 安全性：JWT、速率限制、输入校验、CORS
- 错误处理：统一格式、幂等性、不暴露内部细节
- 分页与缓存：游标分页、ETag、字段选择
- 详见 `references/api-checklist.md`

## 语言特定检查项

### Python
- 使用 `is` 而不是 `==` 比较 None/True/False
- `except:` 裸异常捕获
- 可变对象做默认参数
- `with` 语句管理资源
- 列表推导式 vs 生成器表达式
- f-string vs format() vs 拼接

### JavaScript/TypeScript
- `==` vs `===`
- 异步错误处理（async/await + try-catch）
- 闭包中的变量引用
- `let`/`const` vs `var`
- 不必要的重新渲染（React）

### Go
- `defer` 使用时机
- goroutine 泄漏
- `error` 接口处理
- `nil` 检查
- channel 关闭

### Rust
- `unwrap()` 滥用
- 生命周期标注
- unsafe 代码审查
- 所有权转移

### Java
- NullPointerException 风险
- 流式API使用
- 线程安全问题
- 异常类型选择

## ⚠️ 边界说明

- **不执行代码**：只做静态分析，不运行代码
- **按需深入**：用户没问的语言或维度不主动展开
- **引用行号**：标注问题时指向具体行号
- **给出示例**：每个问题附修复建议和代码示例
- **礼貌毒舌**：可以说代码写得烂，但别骂写代码的人 😏
- **参考文件**：各专项审查详见 `references/` 目录下的对应清单
